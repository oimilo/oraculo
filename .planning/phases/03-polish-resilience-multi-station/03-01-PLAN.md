    6. In `src/machines/oracleMachine.ts`, add a top-level `after` or use the `on` event pattern to implement inactivity reset. The approach: add a global `INACTIVITY_TIMEOUT` event handler. Specifically:
       - Add `after: { 30000: 'IDLE' }` to each narrative state that could "hang" (APRESENTACAO, all NARRATIVA/PERGUNTA/RESPOSTA substates, PARAISO, DEVOLUCAO_*, ENCERRAMENTO).
       - Instead of duplicating, use a more elegant XState v5 approach: add an `ACTIVITY_PING` event handler on all states (via the root machine level) that resets the timer, and use `after` delays.
       - Simplest XState v5 approach: Since `after` delays reset when re-entering a state, we need a different mechanism. The cleanest approach is to use `invoke` with a callback that sets a 30s timer at the root level. But since each state transition already resets state, we can simply add `after: { 30000: { target: 'IDLE', actions: assign(INITIAL_CONTEXT) } }` to the root level of the machine (which covers all states).
       - Actually, in XState v5, root-level `after` applies only if the machine stays in the initial state. Better approach: add `after: { 30000: '#oracle.IDLE' }` to each leaf state EXCEPT IDLE and FIM (which already has its own 5s timer). This is repetitive but explicit and reliable.
       - Most practical: wrap the states that need it. Add the `after` delay to: APRESENTACAO, PARAISO, ENCERRAMENTO, and to each of the compound states INFERNO, PURGATORIO_A, PURGATORIO_B (which covers their substates), and DEVOLUCAO_A_FICAR, DEVOLUCAO_A_EMBORA, DEVOLUCAO_B_PISAR, DEVOLUCAO_B_CONTORNAR.
       - The `after` delay format:
         ```typescript
         after: {
           30000: {
             target: '#oracle.IDLE',
             actions: assign({
               sessionId: '',
               choice1: null,
               choice2: null,
               fallbackCount: 0,
               currentPhase: 'APRESENTACAO',
             }),
           },
         },
         ```
       - Note: This means if any state (speaking, waiting, etc.) has no transition for 30 seconds, it resets. The `after` delay in XState resets every time the state is entered, so normal flow (state transitions happening every few seconds) will never trigger it. Only a truly stuck session hits 30s in one state.
